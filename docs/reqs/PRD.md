# Product Requirements Document (PRD)
## ProfoChatBot - Платформа для интерактивных обучающих курсов в Telegram

**Версия:** 1.0  
**Дата:** 2024  
**Статус:** Активная разработка

---

## 1. Обзор продукта

### 1.1 Описание продукта

ProfoChatBot — это платформа для создания и проведения интерактивных обучающих курсов через Telegram ботов. Система позволяет создавать структурированные курсы с различными типами элементов (сообщения, вопросы, диалоги с ИИ, тесты, мини-приложения), управлять доступом пользователей, отслеживать прогресс и анализировать результаты обучения.

### 1.2 Целевая аудитория

**Основные пользователи:**
- **Создатели курсов** — преподаватели, тренеры, образовательные организации
- **Учащиеся** — пользователи Telegram, проходящие курсы через бота
- **Администраторы** — управление системой, доступом и аналитикой

**Вторичные пользователи:**
- Разработчики, интегрирующие систему через API
- Аналитики, изучающие данные о прохождении курсов

### 1.3 Цели продукта

1. **Образовательные цели:**
   - Предоставить инструмент для создания интерактивных обучающих курсов
   - Обеспечить персонализацию обучения через диалоги с ИИ
   - Поддержать различные форматы обучения (текст, аудио, тесты, практические задания)

2. **Бизнес-цели:**
   - Масштабируемость через мультиботовую архитектуру
   - Контроль доступа и монетизация через restricted курсы
   - Защита от злоупотреблений через систему блокировок

3. **Технические цели:**
   - Простота создания курсов через YAML или веб-интерфейс
   - Надежное хранение данных в PostgreSQL
   - API для интеграции с внешними системами

## 2. Функциональные требования

### 2.1 Управление курсами

#### 2.1.1 Создание курсов

**Описание:** Система должна поддерживать создание курсов двумя способами:
- Через YAML файлы (для разработчиков)
- Через интерактивный генератор в боте (для пользователей без технических навыков)

**Требования:**
- Поддержка формата YAML для описания структуры курса
- Интерактивный генератор курсов через команду `/create`
- Возможность сохранения курсов в базу данных или файловую систему
- Валидация структуры курса при создании

**Приоритет:** Высокий

#### 2.1.2 Редактирование курсов

**Описание:** Создатели курсов должны иметь возможность редактировать существующие курсы.

**Требования:**
- Команда `/edit` для редактирования курса
- Команда `/copy` для создания копии курса
- Сохранение истории изменений (опционально)
- Поддержка редактирования как YAML, так и БД курсов

**Приоритет:** Высокий

#### 2.1.3 Структура курса

**Описание:** Курс состоит из последовательности задач (Tasks) различных типов.

**Требования:**
- Поддержка модульной структуры (элементы вида "Ex8_2", где "Ex8" — модуль)
- Последовательная навигация по задачам (Tasks)
- Возможность навигации между модулями (вперед/назад/к началу модуля)
- Поддержка начального элемента курса
- Task (задача) — минимальная учебная единица, которую студент выполняет внутри группы

**Приоритет:** Высокий

### 2.2 Типы элементов курса

#### 2.2.1 Message (Сообщение)

**Описание:** Простое текстовое сообщение от бота с опциональной кнопкой.

**Требования:**
- Поддержка Markdown и HTML форматирования
- Опциональная кнопка для продолжения
- Поддержка медиа (изображения, видео)
- Настройка превью ссылок

**Приоритет:** Высокий

#### 2.2.2 Audio (Аудио)

**Описание:** Аудиосообщение с возможностью воспроизведения.

**Требования:**
- Поддержка различных аудио форматов (MP3, M4A, OGG)
- URL медиафайлов
- Автоматическое воспроизведение (опционально)

**Приоритет:** Средний

#### 2.2.3 Input (Ввод текста)

**Описание:** Элемент, требующий ввода текста от пользователя.

**Требования:**
- Ожидание текстового ответа от пользователя
- Валидация ввода (опционально)
- Сохранение ответа в историю

**Приоритет:** Высокий

#### 2.2.4 Quiz (Викторина)

**Описание:** Вопрос с одним правильным ответом из нескольких вариантов.

**Требования:**
- Использование Telegram Poll API
- Один правильный ответ
- Автоматическая проверка ответа
- Отображение обратной связи
- Сохранение результата с баллами

**Приоритет:** Высокий

#### 2.2.5 Question (Вопрос)

**Описание:** Опрос без правильного ответа для сбора мнений.

**Требования:**
- Использование Telegram Poll API
- Множественный выбор (опционально)
- Сохранение ответа без оценки

**Приоритет:** Средний

#### 2.2.6 MultiChoice (Множественный выбор)

**Описание:** Вопрос с несколькими правильными ответами.

**Требования:**
- Использование Telegram Poll API
- Множественный выбор ответов
- Проверка всех правильных ответов
- Частичное начисление баллов (опционально)

**Приоритет:** Средний

#### 2.2.7 Dialog (Диалог с ИИ)

**Описание:** Интерактивный диалог с искусственным интеллектом.

**Требования:**
- Интеграция с OpenAI API (GPT модели)
- Поддержка системных промптов
- Настройка параметров модели (temperature, reasoning)
- Поддержка голосовых сообщений с транскрипцией (ElevenLabs)
- Ограничение количества сообщений для контроля расходов
- Автоматическое завершение диалога по условиям

**Приоритет:** Критический

#### 2.2.8 Miniapp (Мини-приложение)

**Описание:** Интеграция с Telegram WebApp для сложных интерактивных заданий.

**Требования:**
- Открытие мини-приложения в Telegram
- Получение результатов через WebApp API
- Сохранение результатов в базу данных
- Поддержка init_data для аутентификации

**Приоритет:** Средний

#### 2.2.9 Test (Итоговый тест)

**Описание:** Финальный тест с подсчетом общего балла.

**Требования:**
- Агрегация баллов из предыдущих элементов
- Отображение итогового результата
- Сохранение финального балла

**Приоритет:** Высокий

#### 2.2.10 Jump (Навигация)

**Описание:** Элемент с опциями навигации по курсу.

**Требования:**
- Кнопки для перехода к различным элементам
- Опция "Понятно" для продолжения
- Опции задержки (wait) для отложенного продолжения

**Приоритет:** Средний

#### 2.2.11 Revision (Повторение)

**Описание:** Повторение элементов с ошибками или случайных элементов.

**Требования:**
- Поиск элементов с score != 1 (ошибки)
- Поиск элементов с score = 1 (для повторения)
- Динамическая генерация списка элементов для повторения
- Интеграция с основным курсом

**Приоритет:** Средний

#### 2.2.12 Delay (Задержка)

**Описание:** Отложенная отправка следующего элемента.

**Требования:**
- Указание времени задержки в формате "2d:3h:45m"
- Сохранение в таблицу waiting_element
- Автоматическая проверка через APScheduler
- Отправка элемента по истечении времени

**Приоритет:** Средний

#### 2.2.13 End (Завершение)

**Описание:** Завершение курса.

**Требования:**
- Отметка курса как завершенного (is_ended = TRUE)
- Финальное сообщение пользователю
- Предотвращение дальнейшего прохождения

**Приоритет:** Высокий

### 2.3 Управление пользователями

#### 2.3.1 Регистрация и начало курса

**Описание:** Пользователь начинает курс через команду `/start` с параметрами пригласительной ссылки группы.

**Требования:**
- Команда `/start group_<group_id>_<token>` для начала курса через пригласительную ссылку группы
- Поддержка UTM параметров через метаданные пригласительной ссылки
- Создание записи в таблице `run` с привязкой к группе и пригласительной ссылке
- Проверка валидности пригласительной ссылки:
  - Существует ли группа и активна ли она
  - Валидна ли пригласительная ссылка
  - Не превышен ли лимит использований (`max_uses`)
  - Не истекла ли ссылка (`expires_at`)
- Проверка доступа для restricted курсов
- Проверка блокировки пользователя
- Автоматическое увеличение счетчика использований пригласительной ссылки

**Приоритет:** Критический

#### 2.3.2 Контроль доступа

**Описание:** Система должна поддерживать ограничение доступа к курсам.

**Требования:**
- Флаг `restricted: yes` в конфигурации курса
- Таблица `courseparticipants` для белого списка пользователей
- Проверка доступа при старте курса
- Отображение сообщения об отказе в доступе

**Приоритет:** Высокий

#### 2.3.3 Система блокировок

**Описание:** Защита от злоупотреблений через автоматическую блокировку.

**Требования:**
- Подсчет сообщений типа `*_chat*` (диалоги с ИИ)
- Автоматическая блокировка при превышении лимита
- Настройка лимита через конфигурацию
- Исключения для определенных пользователей
- Ручное управление блокировками через поле `excluded`
- Периодическая проверка через APScheduler

**Приоритет:** Высокий

### 2.4 Отслеживание прогресса

#### 2.4.1 История взаимодействий

**Описание:** Все взаимодействия пользователя с курсом сохраняются в базу данных.

**Требования:**
- Таблица `conversation` для хранения всех элементов
- Сохранение роли (user/assistant)
- Сохранение JSON данных элемента
- Сохранение ответов пользователя
- Сохранение баллов и максимальных баллов
- Временные метки всех взаимодействий

**Приоритет:** Критический

#### 2.4.2 Сессии прохождения

**Описание:** Каждое прохождение курса создает отдельную сессию (run).

**Требования:**
- Таблица `run` для отслеживания сессий
- Связь всех элементов с run_id
- Хранение UTM меток
- Отметка завершения курса
- Поддержка нескольких сессий одного пользователя

**Приоритет:** Критический

#### 2.4.3 Получение текущего элемента

**Описание:** Система должна отслеживать текущую позицию пользователя в курсе.

**Требования:**
- Функция `get_current_element()` для получения последнего элемента
- Восстановление состояния элемента из JSON
- Поддержка возобновления курса после перерыва

**Приоритет:** Критический

### 2.5 API для интеграции

#### 2.5.1 REST API (FastAPI)

**Описание:** RESTful API для интеграции с внешними системами.

**Требования:**
- FastAPI фреймворк
- Endpoints для управления курсами
- Endpoints для начала курса и получения элементов
- Endpoints для отправки ответов
- Endpoints для получения отчетов
- CORS поддержка
- Валидация через Pydantic модели

**Основные endpoints:**
- `GET /api/courses` — список курсов
- `GET /api/courses/{course_id}` — информация о курсе
- `POST /api/courses/{course_id}/start` — начать курс
- `GET /api/courses/{course_id}/current` — текущий элемент
- `POST /api/courses/{course_id}/answer` — отправить ответ
- `GET /api/reports` — отчеты по курсам

**Приоритет:** Высокий

#### 2.5.2 WebApp интеграция

**Описание:** Поддержка Telegram WebApp для мини-приложений.

**Требования:**
- Валидация init_data от Telegram
- Отправка результатов обратно в бот
- Сохранение результатов в базу данных

**Приоритет:** Средний

### 2.6 Веб-интерфейс для отчетов

#### 2.6.1 Отчеты по курсам

**Описание:** Flask веб-приложение для просмотра данных о прохождении курсов.

**Требования:**
- Страница с таблицей всех взаимодействий
- Фильтрация по username
- Сортировка по различным полям
- Пагинация результатов
- Экспорт в CSV
- Визуализация данных (опционально)

**Приоритет:** Средний

#### 2.6.2 Статистика

**Описание:** Аналитика по курсам и пользователям.

**Требования:**
- Количество пользователей по курсам
- Процент завершения курсов
- Средние баллы
- Распределение по типам элементов

**Приоритет:** Низкий

### 2.7 Мультиботовая архитектура

**Описание:** Поддержка нескольких ботов на одной базе данных.

**Требования:**
- Поле `botname` в ключевых таблицах (legacy, для обратной совместимости)
- Изоляция данных между ботами
- Настройка BOT_NAME через переменные окружения
- Поддержка разных конфигураций для разных ботов
- Связь Бот ↔ Курс осуществляется только через Группы (Group)
- Один бот может иметь несколько групп с разными курсами
- Один курс может быть в нескольких группах на одном боте

**Приоритет:** Высокий

### 2.8 Группы

**Описание:** Группа — это контейнер исполнения курса, объединяющий бота, курс, студентов и опциональное расписание. Группа является центральной сущностью, через которую происходит связь между ботами и курсами.

#### 2.8.1 Создание группы

**Описание:** Создатели курсов могут создавать группы для организации студентов.

**Требования:**
- Создание группы через выбор Бота и Курса
- Указание названия и описания группы
- Автоматическое создание первой пригласительной ссылки (опционально)
- Возможность добавления расписания при создании (опционально)
- Группа связывает один Бот с одним Курсом
- Один бот может иметь несколько групп с разными курсами

**Приоритет:** Высокий

#### 2.8.2 Пригласительные ссылки

**Описание:** Каждая группа имеет одну или несколько пригласительных ссылок для набора студентов.

**Требования:**
- Создание пригласительной ссылки для группы
- Опциональный лимит использования (`max_uses`)
- Опциональный срок действия (`expires_at`)
- Автоматическое увеличение счетчика использований при переходе по ссылке
- Формат ссылки: `https://t.me/<bot_username>?start=group_<group_id>_<token>`
- При переходе по ссылке пользователь автоматически записывается в группу
- Возможность создания нескольких ссылок для одной группы (например, для разных источников)

**Приоритет:** Высокий

#### 2.8.3 Расписание группы

**Описание:** Опциональное расписание для управления временем выхода задач (Tasks).

**Требования:**
- Возможность создания расписания для группы
- Типы расписания:
  - Еженедельное (например, новая задача каждый понедельник в 9:00)
  - Ежедневное (например, новая задача каждый день в 9:00)
  - Кастомное (задачи выходят в определенные даты)
- При наличии расписания задачи выходят в назначенное время
- При отсутствии расписания задачи идут последовательно без пауз
- Использование планировщика (APScheduler) для проверки расписания

**Приоритет:** Средний

- Ограничение доступа к административным функциям
- Защита API ключей через переменные окружения

**Приоритет:** Критический

### 3.4 Масштабируемость

**Требования:**
- Поддержка горизонтального масштабирования
- Возможность запуска нескольких инстансов бота

- Оптимизация запросов к БД

**Приоритет:** Средний

### 3.5 Удобство использования

**Требования:**
- Простой синтаксис YAML для создания курсов
- Интуитивный интерфейс генератора курсов
- Понятные сообщения об ошибках
- Документация для создателей курсов

**Приоритет:** Высокий

## 4. Технические требования

### 4.1 Технологический стек

**Backend:**
- Python 3.12+
- aiogram 3.x (Telegram Bot Framework)
- FastAPI (REST API)
- Flask (веб-отчеты)
- PostgreSQL (база данных)
- psycopg2 (драйвер БД)

**Внешние сервисы:**
- OpenAI API (GPT модели)
- ElevenLabs API (транскрипция голоса)
- Telegram Bot API

**Библиотеки:**
- PyYAML (работа с YAML)
- APScheduler (планировщик задач)
- python-dotenv (переменные окружения)

### 4.2 Архитектура

**Компоненты:**
1. **main.py** — основной Telegram бот (aiogram)
2. **api.py** — REST API (FastAPI)
3. **webreport.py** — веб-интерфейс отчетов (Flask)
4. **db.py** — слой работы с базой данных
5. **course.py** — логика работы с курсами
6. **chat.py** — интеграция с OpenAI API
7. **generator.py** — генератор курсов
8. **editor.py** — редактор курсов
9. **elements/** — типы элементов курса

### 4.3 База данных

**Схема:**
- `account` — аккаунты (тенанты) в multi-tenant архитектуре
- `account_member` — участники аккаунта (создатели/админы)
- `bot` — Telegram-боты
- `course` — метаданные курсов
- `course_element` — элементы курсов из БД (Tasks)
- `group` — группы (контейнеры исполнения курса)
- `invite_link` — пригласительные ссылки для групп
- `schedule` — расписания для групп
- `run` — сессии прохождения курсов (привязаны к группе)
- `conversation` — история взаимодействий
- `waiting_element` — элементы с задержкой
- `courseparticipants` — контроль доступа
- `bannedparticipants` — блокировки пользователей
- `gen_settings` — общие настройки

**Ключевые изменения:**
- Связь Бот ↔ Курс теперь только через `group` (не напрямую)
- `run` привязан к `group` и `invite_link` вместо `course_deployment` и `enrollment_token`
- `course_element` содержит Tasks (бывшие Sections)

Подробное описание структуры БД см. в `docs/database.md` и `docs/reqs/database_schema_saas.md`.

### 4.4 Конфигурация

**Файлы конфигурации:**
- `.env` — переменные окружения (DATABASE_URL, BOT_API_TOKEN, BOT_NAME)
- `config.yaml` — настройки OpenAI, планировщика, блокировок
- `scripts/{bot_folder}/courses.yml` — список доступных курсов
- `scripts/{bot_folder}/*.yaml` — файлы курсов

## 5. Пользовательские сценарии

### 5.1 Сценарий: Создание курса

**Актор:** Создатель курса

**Шаги:**
1. Создатель отправляет команду `/create` в бот
2. Бот запускает интерактивный генератор
3. Создатель отвечает на вопросы генератора
4. Генератор создает YAML файл курса
5. Курс сохраняется в базу данных или файловую систему
6. Курс становится доступным для пользователей

**Результат:** Новый курс создан и готов к использованию

### 5.2 Сценарий: Прохождение курса

**Актор:** Учащийся

**Шаги:**
1. Учащийся переходит по ссылке с параметром `course_id`
2. Открывается Telegram бот с командой `/start course_id`
3. Бот проверяет доступ и создает сессию (run)
4. Бот отправляет первый элемент курса
5. Учащийся взаимодействует с элементами (отвечает на вопросы, общается с ИИ)
6. Прогресс сохраняется в базу данных
7. После завершения курса отображается финальное сообщение

**Результат:** Курс пройден, результаты сохранены

### 5.3 Сценарий: Диалог с ИИ

**Актор:** Учащийся

**Шаги:**
1. Учащийся достигает элемента типа "dialog"
2. Бот отправляет системный промпт и приветственное сообщение
3. Учащийся отправляет текстовое или голосовое сообщение
4. Голосовое сообщение транскрибируется через ElevenLabs
5. Текст отправляется в OpenAI API
6. Бот получает ответ от ИИ и отправляет его учащемуся
7. Диалог продолжается до условия завершения
8. После завершения диалога курс продолжается

**Результат:** Интерактивный диалог с ИИ завершен

### 5.4 Сценарий: Просмотр отчетов

**Актор:** Администратор

**Шаги:**
1. Администратор открывает веб-интерфейс отчетов
2. Выбирает фильтры (username, курс, дата)
3. Просматривает таблицу взаимодействий
4. Экспортирует данные в CSV при необходимости
5. Анализирует статистику по курсам

**Результат:** Получена аналитика по прохождению курсов

## 10. Глоссарий

- **Элемент курса** — единица контента в курсе (сообщение, вопрос, диалог и т.д.)
- **Task (Задача)** — минимальная учебная единица, которую студент выполняет внутри группы (бывшая Section)
- **Group (Группа)** — контейнер исполнения курса, объединяющий бота, курс, студентов и опциональное расписание
- **Invite Link (Пригласительная ссылка)** — ссылка для записи студентов в группу с опциональным лимитом использований
- **Schedule (Расписание)** — опциональное расписание для управления временем выхода задач в группе
- **Run** — сессия прохождения курса пользователем в рамках группы
- **Conversation** — запись взаимодействия пользователя с элементом
- **Dialog** — элемент типа диалог с ИИ
- **Miniapp** — Telegram WebApp мини-приложение
- **Restricted курс** — курс с ограниченным доступом
- **UTM метки** — параметры для отслеживания источников трафика
- **Botname** — идентификатор бота в мультиботовой архитектуре (legacy, для обратной совместимости)
- **Element ID** — уникальный идентификатор элемента (например, "Ex1_2")
- **Module** — группа элементов курса (часть element_id до подчеркивания)
