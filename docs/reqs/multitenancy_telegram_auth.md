# Multitenancy + роли + авторизация через Telegram
**Версия:** 1.1**Дата:** 2026-01-18**Статус:** Техническая спецификация для реализации**Стек:** Next.js 14 (App Router), FastAPI, PostgreSQL, Telegram Bot

# 1. Цели
### 1.1 Бизнес-цели
* Поддержать мультитенантную модель: один пользователь может работать в нескольких аккаунтах (организациях).
* Обеспечить удобный доступ для разных типов пользователей:
  * владелец аккаунта,
  * админы,
  * преподаватели,
  * методисты (instructional designers),
  * при необходимости — обычные участники (member).
* Обеспечить безопасный глобальный доступ для разработчика/поддержки (super_admin).

⠀1.2 Технические цели
* Перейти на авторизацию через **Telegram** как основной способ логина.
* Использовать уже существующую мультитенантную схему с таблицами account и account_member.
* Хранить все бизнес-данные изолированно по account_id.
* Закодировать текущий account_id в сессии/токене, а не в коде (getAccountId() перестает возвращать жёстко 1).
* Добавить ролевую модель на уровне аккаунта и глобальную роль super_admin.

⠀1.3 Ограничения и MVP
* На текущем MVP используется **один аккаунт** с account_id = 1 (тестовый аккаунт).
* Multitenancy на уровне базы данных уже есть, но UI и часть логики пока работают как «один аккаунт».
* Роли:
  * owner, admin, teacher, instructional_designer (ID), опционально member.
  * viewer не используется.
* Преподаватель (teacher) в рамках аккаунта имеет доступ **ко всем группам аккаунта** (школы небольшие, нет смысла дробить доступ по группам).
* Логин по старой модели (email/пароль / ML-паспорт) в новой архитектуре **не используется** (может остаться только для внутренних тестов, за флагом).

# 2. Модель данных (логический уровень)
### 2.1 Таблицаusers — идентичность пользователя
### users — глобальная таблица идентичностей (одна запись на Telegram-пользователя, плюс возможные legacy-пользователи).
**Ключевые поля:**
* user_id INT PRIMARY KEY — внутренний идентификатор пользователя.
* telegram_user_id BIGINT UNIQUE — Telegram ID (основной идентификатор, обязателен для новых пользователей).
* telegram_username TEXT — username из Telegram.
* first_name TEXT
* last_name TEXT
* language_code TEXT — язык интерфейса Telegram.
* photo_url TEXT — ссылка на аватар (опционально).
* last_login_at TIMESTAMPTZ — дата/время последней успешной авторизации.
* is_super_admin BOOLEAN NOT NULL DEFAULT FALSE — глобальная роль разработчика/поддержки.

⠀Legacy-поля (если нужны):
* email TEXT
* password_hash TEXT
* и др. — могут использоваться только для внутризагонных задач, но не являются частью новой нормальной auth-модели.

⠀2.2 Таблица account — аккаунт (организация)
Уже существует. Используется как «тенант»:
* account_id INT PRIMARY KEY
* name TEXT
* slug TEXT UNIQUE
* plan TEXT — тарифный план.
* is_active BOOLEAN
* settings JSONB
* created_at, updated_at и др.

⠀2.3 Таблица account_member — участник аккаунта
Определяет участие пользователя в аккаунте и его роль.
**Ключевые поля:**
* account_member_id INT PRIMARY KEY
* account_id INT NOT NULL REFERENCES account(account_id)
* user_id INT NOT NULL REFERENCES users(user_id)
* role TEXT NOT NULL
* is_active BOOLEAN NOT NULL DEFAULT TRUE
* created_at TIMESTAMPTZ
* last_login_at TIMESTAMPTZ (опционально, дублирует активность внутри аккаунта)

⠀**Уникальное ограничение:**
* UNIQUE (account_id, user_id) — один пользователь может быть участником аккаунта не более одного раза.

⠀**Допустимые значения** role**:**
* 'owner'
* 'admin'
* 'teacher'
* 'instructional_designer'
* 'member' (опционально; может не использоваться в MVP, но зарезервирован для будущего)

⠀Можно реализовать через CHECK или enum-тип.
### 2.4 Другие таблицы (мультитенантные)
Все существующие бизнес-сущности продолжают иметь account_id и изолируются по нему:
* bot
* course
* course_group
* invite_link
* schedule
* run
* conversation
* waiting_element
* courseparticipants и др.

⠀Правило:**любая выборка/модификация данных в этих таблицах должна фильтроваться по** account_id**, взятому из текущей сессии.**

# 3. Модель ролей и прав доступа
Ролевая модель делится на:
**1** **Глобальный уровень (вся система)** — флаг users.is_super_admin.
**2** **Уровень аккаунта** — account_member.role.

⠀3.1 Глобальная роль super_admin
### users.is_super_admin = TRUE
* Имеет технический доступ ко всем account_id и их данным.
* Может:
  * видеть список всех аккаунтов;
  * переключаться в контекст любого аккаунта (impersonation или спец-панель);
  * смотреть и дебажить любые сущности (курсы, группы, ботов, run’ы, логи и т.д.).
* Используется для разработки, диагностики и поддержки.

⠀Во всех проверках доступа:
1 Сначала проверяем: если is_super_admin == TRUE, даём доступ без дальнейших ограничений (кроме явных блоков, если появятся).
2 Если нет — применяем обычные правила по account_id и account_member.role.

⠀3.2 Роли на уровне аккаунта
**3.2.1 Owner**
* Полный контроль над аккаунтом.
* Права:
  * Настройки аккаунта: изменение имени, slug, плана (тариф), общих настроек.
  * Управление участниками:
    * добавление/удаление участников;
    * смена ролей;
    * назначение других owners (если допускается);
    * невозможность «самоликвидироваться» без передачи ownership.
  * Управление всеми сущностями:
    * боты, курсы, группы, расписания, invites, участники групп, аналитика.
  * Удаление аккаунта (при подтверждении).

⠀**3.2.2 Admin**
* Администратор аккаунта (почти всё, кроме критичных операций с аккаунтом).

⠀Права:
* Боты:
  * создание/редактирование/удаление ботов;
  * настройка токенов и параметров бота.
* Курсы:
  * создание/редактирование/удаление курсов;
  * связка курсов с ботами и группами.
* Группы:
  * создание/редактирование/удаление групп;
  * назначение курсов и ботов для групп;
  * управление расписанием (schedule) групп.
* Участники аккаунта:
  * приглашать новых участников в аккаунт;
  * менять им роли (кроме поднятия себя до owner и, возможно, прав на owner в целом, это бизнес-политика).
* Аналитика:
  * полный доступ ко всей статистике аккаунта.

⠀Не может (по сравнению с owner):
* удалять аккаунт целиком;
* менять критичные биллинговые настройки, если они отданы только owner’у;
* трогать глобальные системные вещи.

⠀**3.2.3 Teacher (преподаватель)**
* Работает с **группами и студентами**.
* В малых школах — имеет доступ ко всем группам **внутри аккаунта**, без доп. ограничений по группам.

⠀Права:
* Группы:
  * видеть список **всех** групп аккаунта;
  * открывать детали группы:
    * список участников,
    * прогресс,
    * выполненные задания,
    * статистику по группе.
* Студенты:
  * видеть индивидуальный прогресс студента в контексте группы;
  * просматривать ответы, домашние задания и результаты тестов;
  * добавлять комментарии/фидбек (когда появится этот функционал);
  * инициировать отправку сообщений группе / отдельным студентам (например, напоминания).

⠀Ограничения:
* Не может:
  * создавать/удалять/редактировать ботов;
  * создавать/удалять/редактировать курсы;
  * менять настройки аккаунта;
  * управлять ролями и участниками аккаунта;
  * создавать/удалять группы;
  * изменять расписание (schedule) групп (если не решено иначе; по умолчанию schedule — зона admin).

⠀Авторизация (резюме):
* Фильтрация по account_id (как для всех);
* Проверка: role IN ('owner', 'admin', 'teacher') даёт доступ к разделу /groups.
* Для операций редактирования групп: role IN ('owner', 'admin').

⠀**3.2.4 Instructional designer (ID)**
* Ответственность: **курсы и их содержимое**.

⠀Права:
* Курсы:
  * создание курсов;
  * редактирование структуры курса;
  * управление YAML-описанием курсов;
  * управление версиями курса (если предусмотрено).
* Аналитика по курсам:
  * видеть агрегированную статистику по курсу (успешность, точку отвалов) в разрезе всего аккаунта.

⠀Ограничения:
* Не может:
  * создавать/редактировать/удалять ботов;
  * создавать/редактировать/удалять группы;
  * управлять расписанием групп;
  * управлять участниками аккаунта;
  * управлять ролями.

⠀Авторизация:
* Доступ к /courses и /course-editor:
  * role IN ('owner', 'admin', 'instructional_designer').
* Mutation’ы по курсам:
  * role IN ('owner', 'admin', 'instructional_designer').

⠀**3.2.5 Member (опционально)**
* Базовая роль «обычный сотрудник».
* MVP может **не использовать** его явно, но тип роли лучше заложить.
* Типичный сценарий:
  * read-only доступ к некоторой аналитике,
  * доступ к отчётам без права редактирования сущностей.
