# Развертывание на Продакшене

Этот документ описывает, как устроено боевое окружение веб-приложения, и содержит инструкции по деплою свежего кода на продакшен-сервер.

## Разница конфигураций (Prod vs Local)

Важно понимать ключевую разницу в файлах `docker-compose.yml`:
1. **Разработка (Local):** `webapp/docker-compose.yml` (запускается через `make webapp-docker-up`). Этот конфиг монтирует локальные папки (`volumes`) и ждет внешний `.env` для переменных.
2. **Боевой сервер (Prod):** корневой `/docker-compose.yml` (запускается `cd /var/www/profochatweb && docker-compose up -d`). Этот конфиг **зашивает исходный код в образы** на этапе сборки и пробрасывает боевые порты:
   - `3300:3000` для фронтенда (на 3300 смотрит Nginx сервера)
   - `8300:8000` для бэкенда

## 1. Автоматизированный деплой файлов

Для того чтобы не копировать файлы вручную и не тянуть весь репозиторий на сервер, в проекте есть bash-скрипты для выборочного деплоя измененных файлов поверх SSH.

### Шаг 1. Сбор измененных файлов
Когда вы закончили работу и сделали коммит(ы), вам нужно собрать список изменившихся файлов относительно нужного коммита (например, предыдущего состояния на проде):

```bash
sh bin/utils/_get_changes.sh <COMMIT_HASH>
```
*Где `<COMMIT_HASH>` — это хеш (например, `896fac998`), ПОСЛЕ которого были изменения, которые вы хотите задеплоить.*

Скрипт автоматически пройдется по этим коммитам и сохранит нужные файлы в `changed.txt` (он также автоматически игнорирует ненужные папки типа `docs/`).

### Шаг 2. Копирование файлов на сервер
Сразу после формирования списка запустите второй скрипт:

```bash
sh bin/utils/_deploy.sh
```
Скрипт возьмет файлы из `changed.txt`, упакует их, подключится к серверу `profoweb@scrumtrek.ru` и распакует их прямиком в `/var/www/profochatweb`.
*(Вам может потребоваться ввести пароль от вашего SSH-ключа)*.

## 2. Применение изменений на сервере (Пересборка)

Так как боевой `docker-compose.yml` не использует `volumes`, простое копирование файлов не обновит запущенное приложение (Next.js должен полностью пересобрать страницы на уровне `npm run build`).

Чтобы применить ваши скопированные файлы, нужно пересобрать Docker-образы.

### Шаг 1. Подключение к серверу
```bash
ssh profoweb@scrumtrek.ru
```

### Шаг 2. Пересборка
Перейдите в корневую папку (именно в ту, где лежит "боевой" `docker-compose.yml`) и выполните команду с флагом `--build`:

```bash
cd /var/www/profochatweb
docker compose up -d --build
```

**Что при этом произойдет:**
- Docker найдет новые файлы исходного кода.
- Используя кеш, он мгновенно пропустит этапы установки зависимостей (если вы не трогали `package.json` / `requirements.txt`).
- Он пересоберет слои с кодом и запустит финальный `npm run build` для фронта.
- Аккуратно "погасит" старые контейнеры и поднимет новые, свежесобранные.

*Примечание: Делать предварительный `docker compose down` не нужно, `up -d` обновит запущенные контейнеры бесшовно.*

### Решение проблем со старым docker-compose
Если на сервере стоит старая версия `docker-compose` (например, 1.29.2, написанная на Python, отличие в дефисе между docker и compose), при запуске новой сборки вы можете получить ошибку `KeyError: 'ContainerConfig'`. Это известный баг совместимости.

Чтобы его обойти, жестко удалите старые контейнеры перед запуском:
```bash
docker rm -f profochatweb_frontend_1 profochatweb_backend_1
docker-compose up -d --build
```
*(Но лучше используйте новую команду плагина: `docker compose up -d --build` без дефиса).*
