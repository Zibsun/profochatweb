# Локальное развертывание приложения (Lessons Learned)

В этом документе собраны правила и подводные камни при локальном развертывании проекта (в частности, папки `webapp`) с использованием Docker Compose, выявленные опытным путем.

## 1. Запуск Frontend в Docker (и порт 3002)

Frontend представляет собой Next.js приложение.
Изначально Next.js использует порт `3000`. Однако, **почему мы используем 3002 порт?**
Если вы заглянете в `bin/webapp/dev-all.sh`, вы увидите, что скрипты разработчиков явно ожидают запуск фронтенда на `3002`. Это сделано для избежания конфликтов портов на локальной машине разработчика (так как в корне проекта есть еще один docker-compose.yml, который тоже может занимать порты). В `package.json` веб-приложения жестко прописано использование `3002`:

Для того чтобы Next.js корректно отдавал проект наружу из Docker-контейнера, в `package.json` должно быть явно указано `-H 0.0.0.0` (прослушивание всех интерфейсов):
```json
"scripts": {
  "dev": "next dev -p 3002 -H 0.0.0.0",
  // ...
}
```

Также в блоке `docker-compose.yml` фронтенд должен пробрасывать именно этот порт (`3002:3002`):
```yaml
frontend:
  ports:
    - "3002:3002"
  environment:
    NEXT_PUBLIC_API_URL: http://localhost:8000
```

## 2. Два подхода к настройке `docker-compose.yml` (База данных и .env)

В проекте исторически есть закоммиченная версия `webapp/docker-compose.yml`, которая работает «из коробки» без ссылок на `.env` файлы. Важно понимать разницу между ней и тем вариантом, когда мы подключаемся к существующей/удалённой БД. Оба варианта имеют место быть:

### Вариант А: "Всё в коробке" (Как в закоммиченной версии)
Этот вариант использует Асхат. В нем `docker-compose.yml`:
1. Запускает собственный изолированный контейнер базы данных (`service: db`, PostgreSQL 14).
2. Бэкенд жестко ссылается на этот контейнер (`DATABASE_URL: postgresql://profochatbot:profochatbot@db:5432/profochatbot_web`).
3. Никакие внешние `.env` файлы не читаются. Ключи (например, `OPENAI_API_KEY`) передаются "на лету" из терминала разработчика (через `${OPENAI_API_KEY}`).
*Плюс:* Не нужно ничего настраивать, база поднимается пустая и локальная. Идеально для старта с нуля.

### Вариант Б: Работа с внешней базой и `.env` файлом
Этот вариант использует Алексей.
Если вы хотите запускать приложение локально, но подключаться к **существующей удалённой** базе данных (например, `37.230.119.106`), вам нужно переопределить `DATABASE_URL`. И чтобы не мучиться с переменными в консоли `docker-compose.yml` модифицируется для чтения настроек из `.env`:

```yaml
backend:
  env_file:
    - ./backend/.env
```

Внутри `webapp/backend/.env` обязательно должны присутствовать:
- `DATABASE_URL` — ссылка на базу
- `SECRET_KEY` — ключ для шифрования
- `OPENAI_API_KEY` — ключ OpenAI
- `OPENAI_BASE_URL` — (если нужен прокси)

> **Важно:** при использовании **Варианта Б** убедитесь, что вы не дублируете жестко-закодированные переменные в `environment:` секции docker-compose, так как они перебьют значения из `.env` файла.

## 3. Инициализация базы данных

Если вы подключаетесь к внешней БД (или удаленной), где еще нет таблиц от веб-версии, их придется создать.

Так как `alembic` в `webapp/backend` не настроен должным образом (пустой `alembic.ini`), выполнить `alembic upgrade head` не удастся.

Создать таблицы для MVP можно напрямую через SQLAlchemy скриптом изнутри контейнера (исключив при этом устаревшие модели курсов типа `course_deployment`, которые могут искать несуществующую таблицу `bot`):
```bash
docker compose exec backend python -c "
from app.database import Base, engine
from app.models import (user, chat_session, chat_message, quiz_attempt,
    course_progress, lesson, lesson_step, account, account_member)

for table in Base.metadata.tables.values():
    try:
        table.create(bind=engine, checkfirst=True)
    except Exception as e:
        print(f'Skipped {table.name}: {e}')
"
```

## 4. Доступ к курсам (YAML файлам)

Бэкенд читает курсы напрямую из файлов (например, `scripts/courses.yml` и файлов самих курсов).
Обратите внимание, что Docker-контейнер `backend`, собранный из `webapp/backend`, изначально **не видит содержимого корневой директории проекта** (там, где лежат папки `scripts/` и `elements/`). 

Поэтому в `webapp/docker-compose.yml` необходимо смонтировать корень проекта и передать переменную окружения `PROJECT_ROOT`, чтобы скрипт знал, где искать `scripts/`:
```yaml
backend:
  environment:
    PROJECT_ROOT: /project
  volumes:
    - ./backend:/app
    - ../:/project:ro
```

## Итоговая схема запуска:

1. Настроить `webapp/backend/.env`.
2. Убедиться, что скрипты запуска `Next.js` и порты в `docker-compose.yml` согласованы.
3. Из папки `webapp` выполнить `docker compose up -d`.
4. (Опционально) Инициализировать таблицы в БД.
5. Открыть Frontend по адресу `http://localhost:3002` (или другому, указанному в конфигурации).
